- Navigate first to the Support website: `http://10.80.171.1`:
![description](./img/Pasted%20image%2020260724131953.png)
- There is not much to observe here and traditionally insecure endpoints like `/assets` and `/sitemap.xml` are inaccessible
	- Thus, move onto Dirbuster to enumerate endpoints and new content discovery
- Run Dirbuster via: `dirbuster http://10.80.171.1`
![description](./img/Pasted%20image%2020260723172115.png)
- The small Dirbuster wordlist was chosen, with only brute forcing dirs and files ending in `.php` due to the scope of this challenge
- Output:
![description](./img/Pasted%20image%2020260723172036.png)
- Once again, almost all of these endpoints are inaccessible, meaning we will need to login first
	- However these are crucial findings which will come into play later

- Going back to the login page, `help@support.thm` is seemingly a real user
	- This is due to the placeholder in the username field as well as the text at the bottom reading `Problems signing in? Contact IT Operations @ help@support.thm`
	- Brute forcing this account will be a possible attack vector if the website lacks rate limiting or account lockouts
- Test for rate limiting:
```bash
seq 1 50 | xargs -P 50 -I{} curl -s -o /dev/null -w "%{http_code}\n" http://10.80.171.1/?email=help%40support.thm&password=a
```
- Output:
![description](./img/Pasted%20image%2020260724132658.png)
- If all 50 of these requests returned `200`, this means there is no rate limiting or account lockout and we are free to attempt brute forcing.
- To do this, first find the size of an incorrect login response:
	- This is because we want to filter out incorrect responses, and we can do so by filtering out responses that match the size of an incorrect login (assuming a successful login will be different):
```bash
ffuf -w test.txt -X POST -d "email=test.txt&password=a" -H "Content-Type: application/x-www-form-urlencoded" -u http://10.80.171.1
```
- Output:
![description](./img/Pasted%20image%2020260723161006.png)
- The size field, `2678`, is what we need. We can now craft our input:
```bash
ffuf -w /usr/share/wordlists/rockyou.txt -X POST -d "email=help@support.thm&password=FUZZ" -H "Content-Type: application/x-www-form-urlencoded" -u http://10.80.171.1 -fs 2678
```
- Explanation:
	- `ffuf` initiates the `ffuf` program
	- `-w` refers to the path to our wordlist, in this case `rockyou.txt`
	- `-X` marks the HTTP method, in this case `POST`
	- `-d` is the data payload for the URL
		- `email` is `help@support.thm`
		- `password` is `FUZZ` which defaults to the wordlist we specified
	- `-H` is to URL encode the data
	- `-u` is the target URL
	- `-fs` filters out incorrect logins from output via response size, which we figured out earlier was `2678`
- Output:
![description](./img/Pasted%20image%2020260723155514.png)
- `snoopy` was flagged as the correct password, we can now successfully login as `help@support.thm`:
![description](./img/Pasted%20image%2020260723161703.png)

**Flag 2:**
<a id="flag-2"></a>
- Once logged in as admin, the only new feature is the date/time widget near the bottom of the page:
![description](./img/Pasted%20image%2020260723180711.png)
- Upon closer inspection via the source code, the `date` command is being invoked each time this widget is being selected:
![description](./img/Pasted%20image%2020260723180816.png)
- The `value` field is manipulatable, meaning it is vulnerable as a command injection attack vector
- Open the developer tools (right click -> Inspect) and click the **Network** tab. Then click one of the date/time widgets.
	- The response should appear as a `POST` request in the **Network** tab. Right click it and then left click **Edit and Resend**:
![description](./img/Pasted%20image%2020260723181040.png)
- Scroll down to the request body where we can see the application invoking `date` via `sys=date`:
![description](./img/Pasted%20image%2020260723181114.png)
- Trying to replace `date` with a cat command to the file and clicking **Send** will yield this response:
![description](./img/Pasted%20image%2020260723181237.png)
- To circumvent this filter, add a `date;` before the cat command so that it passes the date command check and executes your next command
	- This will be our payload:
```bash
date; cat /home/ubuntu/user.txt
```
![description](./img/Pasted%20image%2020260723182012.png)
- The flag will then be revealed alongside the date:
![description](./img/Pasted%20image%2020260723181446.png)
- This is the point at which RCE (Remote Code Execution) is achieved. The attacker functionally has a shell running as `www-data` . They can initiate this into a reverse shell via `netcat` and start running arbitrary code on the server.

<a id="retrospective-analysis"></a>
**Retrospective Analysis**
There are a combination of several design flaws within this web application that lead to vulnerabilities:
- No Rate Limiting/Account Lockout.
- Insecure Cookie.
- Insecure API.
- Local File Inclusion.
- Command Injection.
No rate limiting was what allowed the exploitation of the next four design flaws. Due to the lack of rate limiting, anyone is able to initiate brute force attacks on any user as long as they know they are targeting a real, registered username. The insecure cookie is a consequence of assuming hashing values is enough to conceal them. In reality, hash tables of thousands of common words already exist across multiple hashing values, especially a value like `true` and `false`. The insecure API allows information disclosure and access to viewing other users data, extracting the username for the administrator with ease. The LFI demonstrates how loading files as parameter input can be easily abused to retrieve unintended files. And finally, invoking the `date` command with a weak filter was the final step in achieving RCE.

To remediate, rate limiting and/or account lockout mechanisms should immediately be put in place. This will shut down brute forcing credentials as an attack vector and ensure an attacker cannot run programs like `ffuf` to gain unauthorised access. The cookie value should be modified to a securely encrypted algorithm rather than a hash value that can be matched against a hash table. For the insecure API, replace the numeric sequential integer with a UUID (Universally Unique Identifier) so that it is not possible to easily guess and enumerate users. The theme feature should either be reworked to not load `.php` files in order to change the page's theme, or user input should be sufficiently validated client side and sanitised server side. Finally, to eliminate the command injection, there should also be client side validation and server side sanitisation to strip any malicious input of any bad commands.
